
Die Kombination SQL-Query plus zweite Query mit "id1 IN (...)" kann als
sequence query oder als reduction query ausgeführt werden:

- SQ: Der Vorteil SQ vs. RQ ist nur Sinnvoll, wenn die Daten nicht aufgehoben
	(sondern gleich an den Client ausgegeben) werden, da eine SQ ansonsten zur
	RQ degeneriert. Dann ist aber keine zweite Query per IN möglich, weil die
	Results der zweiten Query nicht mehr in die Response eingefügt werden können.
	Ohne IN muss eine Query pro Zeile gesendet werden (langsam).

- RQ: Braucht bei größeren Tabellen viel Speicher. Bei kleineren Tabellen ist
	das sicherlich die sinnvollste Lösung.

Weitere Lösungen für große Tabellen:

- Zeilen Blockweise verarbeiten. Jeweils Zeilen zu einem Block sammeln, zweite
	Query mit IN, dann an den Client ausgeben und weiter mit dem nächsten
	Block. Sicherlich die sinnvollste Variante für riesige Datenmengen.

- SQL-seitig die zweite Query, z.B. als Subquery, ausführen. Weiß ich zuwenig
	drüber (Alex weiß sicher mehr).
